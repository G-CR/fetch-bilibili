# 路线图（当前版本）

本文档用于描述项目当前的阶段判断和中短期演进方向，不再保留早期
“MVP -> 多平台 -> App 化”的概念性设想。

具体任务拆分、执行状态和临时优先级，请以 `docs/todo.md`、
`docs/superpowers/specs/` 和 `docs/superpowers/plans/` 为准；本文档只回答
“项目现在走到哪一步，以及接下来优先往哪几条线继续推进”。

## 1. 当前阶段判断

项目已经完成早期概念验证，当前阶段应定义为：

- 单平台（B 站）单机可运维版本
- 后端服务 + 独立前端控制台的完整闭环版本
- 以稳定运行、恢复能力、人工审核和本地存储管理为核心目标的版本

这意味着当前路线图的重点，不是继续扩平台或扩终端形态，而是围绕现有能力做
稳定性、运维性和使用效率的持续收口。

## 2. 已达到的阶段

### 2.1 基础业务闭环已完成

当前仓库已经具备以下基础闭环：

- 正式博主管理
- 新视频发现与下载
- 视频可访问性检查
- 存储超阈值清理
- 启动恢复与重复任务保护

这部分已经不是“计划中能力”，而是当前版本基线。

### 2.2 单机运维能力已完成首轮闭环

当前版本已具备基础运维能力：

- MySQL 持久化与自动迁移
- Docker / Compose 部署
- 单视频重下、单视频检查、手动 cleanup
- 配置在线读取、写回与保存后重启
- Cookie / 风控运行态观测
- 浏览投影 `library/` 与定时对账

这说明项目已从“能跑”进入“可以长期维护”的阶段。

### 2.3 前端控制台与实时同步已完成首轮闭环

当前版本已具备：

- 独立前端控制台
- 候选池审核页面
- 任务、视频、博主、系统、存储状态查看
- SSE 实时事件流
- 断线重连与补快照纠偏

因此，当前系统不应再被视为“仅后端服务”。

## 3. 当前路线图重点

### 3.1 路线一：单平台内的运维效率优化

在保持“只做 B 站、只做单机”的前提下，继续优化现有系统的可操作性。

重点包括：

- 候选池审核效率与评分解释质量
- 前端控制台的运维操作体验
- 任务失败原因、恢复路径和状态展示清晰度
- 配置修改、系统重启、存储压力等高频场景的可见性

这条路线关注的是“让现有系统更好用”，而不是扩大边界。

### 3.2 路线二：存储与浏览链路继续稳固

当前系统已经有 `store/` 与 `library/` 双层结构，后续仍应优先在现有模式内打
磨，而不是切换到新的存储架构。

重点包括：

- 清理策略与保留规则的可解释性
- 浏览投影对异常退出、事件丢失、人工误删的修复能力
- 存储统计、稀有视频统计和清理结果的一致性
- 本地浏览目录对人工检索场景的支撑能力

### 3.3 路线三：质量与验收闭环补齐

主入口事实文档与单平台稳定性文档的第一轮收口已经完成，后续重点不再是继续
把同一批入口文档长期挂成最高优先级，而是把自动化回归和验收证据补齐。

重点包括：

- 命中路径补测与回归测试补齐
- 覆盖率基线维护
- Shell / PowerShell 部署脚本一致性复验
- 部署、恢复、重启、异常场景的 smoke 验证沉淀

这条路线的目标是减少“文档说得清，但自动化证据不足”的剩余风险。

## 4. 当前不进入路线图的方向

以下方向当前明确不进入主路线图，只保留为远期讨论主题：

- 多平台扩展（抖音、快手、小红书等）
- 移动端或桌面端原生 App
- 多用户、权限、租户隔离
- 云端对象存储、分布式队列或多节点 worker 集群
- 候选池自动转正

这些方向之所以暂不进入路线图，不是因为它们永远不会做，而是因为它们与当
前阶段目标不一致，会显著抬高实现和维护复杂度。

## 5. 当前推荐顺序

若没有更高优先级的新需求插入，建议按以下顺序继续推进：

1. 先处理候选池审核与运维效率优化。
2. 再处理存储与浏览链路的可解释性与一致性增强。
3. 最后持续补齐单平台质量专项、补测和部署脚本一致性复验。

## 6. 维护约定

- 当项目阶段判断发生变化时，才更新本路线图。
- 单个功能、单个任务或单次修复，不要求每次都改本文件。
- 若 `docs/todo.md` 中的执行优先级与本文件方向冲突，应先回到
  `docs/requirements.md` 和当前实现确认边界，再决定是否更新路线图。
