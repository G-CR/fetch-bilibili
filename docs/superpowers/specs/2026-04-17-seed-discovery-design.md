# 种子池自动发现设计规格

## 背景

当前系统已经具备以下能力：

- 维护正式追踪博主列表。
- 定期抓取博主最新视频。
- 对已下载视频执行下架检查、稳定判定与存储清理。
- 通过 Web 驾驶舱管理博主、任务、配置与存储状态。

但系统仍然完全依赖人工录入博主，缺少“自动发现潜在高价值博主”的入口。这会带来几个直接问题：

- 追踪池增长速度受限，容易漏掉生命周期短、删稿概率高的新账号。
- 已有系统只能在“你已经知道要追谁”时发挥作用，无法主动扩展素材来源。
- 正式追踪池与候选池没有边界，无法形成“发现 -> 审核 -> 转正”的闭环。

用户希望系统能够自动筛选出一批值得关注的 B 站博主，进入候选池，由人工审核后再加入正式追踪，从而让绝版视频库的增长更主动、更可持续。

## 目标

- 新增一套与正式追踪链路解耦的“候选博主发现”能力。
- 通过关键词发现与关系扩散两条链路，持续产出候选博主。
- 为每个候选提供可解释的评分与命中原因，而不是黑盒推荐。
- 通过 Web 页面完成候选池的查看、筛选、审核、批准、忽略与拉黑。
- 批准后将候选安全写入现有 `creators`，并可选地触发一次拉取。
- 发现链路失败、超时、限流时，不影响现有 `fetch/download/check/cleanup` 主链路。

## 非目标

- 第一版不做跨平台发现，只覆盖 B 站。
- 第一版不做自动批准，不做“高分即自动转正”。
- 第一版不做机器学习或反馈学习模型。
- 第一版不做复杂作者图谱，不做实时大规模推荐流。
- 第一版不做发现即下载，只负责把候选送入人工审核队列。

## 方案选择

### 方案一：轻量候选池评分器

核心思路：

- 来源采用“关键词发现 + 已追踪博主关系扩散”。
- 系统把发现到的候选统一写入 `candidate_creators`。
- 评分器根据风险内容、活跃度、相似度、删除痕迹等信号打分。
- 前端展示候选池，人工执行“加入追踪 / 忽略 / 拉黑”。

优点：

- 实现成本低，能快速落地第一版。
- 评分可解释，便于人工审核与后续调参。
- 风控边界清晰，不会误触自动下载主链路。

缺点：

- 命中率初期依赖人工配置关键词和权重。
- 关系扩散能力较弱，前期增长速度有限。

### 方案二：带反馈学习的候选池

核心思路：

- 在方案一基础上，记录用户批准/忽略行为，动态修正评分权重。

优点：

- 随着人工反馈积累，候选质量有机会持续提升。

缺点：

- 第一版数据样本不足，容易过拟合。
- 实现和调试复杂度明显更高。

### 方案三：发现图谱引擎

核心思路：

- 基于作者、视频、关键词、话题、相关推荐构建发现网络，持续扩散候选。

优点：

- 上限高，后续扩平台时也更容易复用。

缺点：

- 当前阶段明显过重。
- 会把系统从“工具型服务”推向“推荐系统”，不符合第一版目标。

## 最终决策

采用方案一：`轻量候选池评分器`。

理由：

- 现阶段最重要的是尽快形成“发现 -> 审核 -> 转正”的稳定闭环。
- 第一版需要的是低风险、可解释、可调参，而不是更复杂的智能推荐。
- 在系统积累了真实的候选审核结果与下架样本之前，没有足够依据支撑更重的算法投入。

## 设计原则

### 1. 候选池与正式追踪池严格分离

- `candidate_creators` 只负责候选发现与审核，不直接承担正式追踪责任。
- `creators` 仍然是正式追踪的唯一入口。
- 任何自动发现结果都不能直接触发下载链路。

### 2. 所有候选都必须可解释

- 每个候选必须能回答“为什么推荐它”。
- 分数必须能拆解到来源与因子，不接受纯总分黑盒。
- 前端必须能展示评分明细和来源明细。

### 3. 发现链路失败不能污染主链路

- `discover` 任务和现有任务队列逻辑隔离。
- 发现失败只记录任务失败与日志，不回滚正式追踪数据。
- 发现链路的 QPS、分页、候选上限必须独立配置。

### 4. 第一版保守，避免过度设计

- 只做 B 站。
- 只做关键词发现和轻量关系扩散。
- 只做人工批准，不做自动批准。
- 只做可解释评分，不做在线学习。

## 范围与整体流程

### 范围

第一版只覆盖“B 站候选博主发现”，不涉及跨平台，也不改变现有下载、检查、清理链路的核心行为。

### 新增子系统

建议新增独立的 `seed discovery` 子系统，拆成 4 个职责单元：

- `发现器（Discoverer）`
  负责从关键词和关系扩散入口抓取候选。
- `评分器（Scorer）`
  负责计算候选得分，并写出评分明细。
- `候选池（Candidate Store）`
  负责落库、去重、状态流转和人工反馈沉淀。
- `审核入口（Review UI / API）`
  负责让用户查看、筛选和处理候选。

### 主流程

1. 调度器触发 `discover` 任务。
2. 发现器执行关键词发现。
3. 发现器执行关系扩散发现。
4. 对候选去重并写入候选池。
5. 评分器计算总分与分项原因。
6. 前端候选池页面按分数展示。
7. 用户人工执行 `批准 / 忽略 / 拉黑 / 重新审核`。
8. 批准后把候选写入正式 `creators`，并按配置决定是否立即入队一次 `fetch`。

## 数据模型

第一版新增 3 张表。

### 1. `candidate_creators`

候选博主主表，用于保存候选主体、当前状态和总分。

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint unsigned | 主键 |
| `platform` | varchar(32) | 平台，第一版固定为 `bilibili` |
| `uid` | varchar(64) | 平台博主唯一标识 |
| `name` | varchar(255) | 博主名称 |
| `avatar_url` | varchar(1024) | 头像地址 |
| `profile_url` | varchar(1024) | 空间地址 |
| `follower_count` | bigint unsigned | 粉丝数 |
| `status` | enum | `new / reviewing / approved / ignored / blocked` |
| `score` | int | 当前总分 |
| `score_version` | varchar(32) | 评分公式版本 |
| `last_discovered_at` | datetime | 最近一次被发现时间 |
| `last_scored_at` | datetime | 最近一次评分时间 |
| `approved_at` | datetime nullable | 批准时间 |
| `ignored_at` | datetime nullable | 忽略时间 |
| `blocked_at` | datetime nullable | 拉黑时间 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

约束和索引建议：

- `UNIQUE KEY uk_platform_uid (platform, uid)`
- `KEY idx_status_score (status, score)`
- `KEY idx_last_discovered (last_discovered_at)`

### 2. `candidate_creator_sources`

候选来源表，用于记录候选是如何被发现的。一个候选可以有多个来源。

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint unsigned | 主键 |
| `candidate_creator_id` | bigint unsigned | 关联候选 |
| `source_type` | varchar(32) | `keyword / related_creator / related_video / manual_import` |
| `source_value` | varchar(255) | 关键词文本、来源 UID、视频 ID 等 |
| `source_label` | varchar(255) | 面向前端展示的说明 |
| `weight` | int | 该来源贡献的分值 |
| `detail_json` | json | 扩展细节 |
| `created_at` | datetime | 创建时间 |

索引建议：

- `KEY idx_candidate_source (candidate_creator_id)`
- `KEY idx_source_type_value (source_type, source_value)`
- `UNIQUE KEY uk_candidate_source (candidate_creator_id, source_type, source_value)`

### 3. `candidate_creator_score_details`

评分明细表，用于记录分数构成，支撑可解释性和调参。

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint unsigned | 主键 |
| `candidate_creator_id` | bigint unsigned | 关联候选 |
| `factor_key` | varchar(64) | 评分因子键，例如 `keyword_risk` |
| `factor_label` | varchar(255) | 因子说明 |
| `score_delta` | int | 分值变化 |
| `detail_json` | json | 结构化细节 |
| `created_at` | datetime | 创建时间 |

### 状态流转

建议只允许以下状态流转：

- `new -> reviewing`
- `reviewing -> approved`
- `reviewing -> ignored`
- `reviewing -> blocked`
- `ignored -> reviewing`
- `approved` 保留在候选池中，不再回退

补充规则：

- `ignored` 候选再次被自动发现时，只刷新发现时间、来源和分数，不自动回到 `reviewing`。
- `blocked` 候选再次被自动发现时，直接过滤，不写入候选池。
- `approved` 候选再次被自动发现时，只做元数据补齐，不重复创建正式追踪记录。

### 与现有表的关系

- 批准时把候选写入现有 `creators`。
- 候选记录不删除，用于保留发现历史与人工反馈。
- 如果正式追踪里已经存在同 `platform + uid`，则候选池只更新状态和历史，不重复写入正式追踪。

## 发现流程

第一版采用两条发现链路并行。

### 1. 关键词发现

系统维护一组可配置关键词，例如：

- `影视剪辑`
- `高能片段`
- `删减片段`
- `补档`
- `重传`
- `直播切片`
- `熟肉`
- `未删减`
- `演唱会`
- `MV`
- `赛事集锦`
- `资源合集`

发现逻辑：

1. 调度任务按关键词轮询搜索。
2. 从搜索结果中抽取博主及上下文信息。
3. 对同一博主聚合多个关键词命中。
4. 将命中信息写入 `candidate_creator_sources`。
5. 去重后交给评分器重算总分。

### 2. 关系扩散发现

第一版只做轻量扩散，不引入完整图谱。扩散信号包括：

- 与已追踪博主标题关键词高重合的作者。
- 与已追踪视频处于相同题材/分区的作者。
- 在同一搜索结果页、相关推荐列表中反复共现的作者。

第一版约束：

- 不做无限层级扩散。
- 每轮扩散只从已追踪博主出发一跳。
- 每个来源博主设置最大扩散数量，避免长尾爆炸。

### 3. 去重与刷新

- 发现到已存在候选时，只更新 `last_discovered_at` 与来源信息。
- `approved` 候选再次被发现时，不新建记录。
- `blocked` 候选再次被发现时，直接过滤。
- `ignored` 候选再次被发现时，保留 `ignored` 状态，仅更新元数据和评分。
- 每轮评分前清理并重建该候选的评分明细，避免历史脏数据累积。

## 评分公式

第一版评分公式：

`总分 = 风险内容分 + 活跃度分 + 相似度分 + 删除痕迹分 + 体量修正分 + 人工反馈分`

### 1. 风险内容分

根据标题、简介、分区、来源关键词命中高风险内容类型。

建议初始规则：

- 命中高风险关键词：`+8 ~ +20`
- 累计上限：`40`

### 2. 活跃度分

依据最近 30 天更新数量：

- `1-2` 条：`+5`
- `3-5` 条：`+10`
- `6+` 条：`+15`

### 3. 相似度分

根据与已追踪池的相似程度给分：

- 弱相似：`+5`
- 中相似：`+10`
- 强相似：`+20`

相似度来源：

- 标题词重合度
- 内容分类/分区接近度
- 同搜索页共现强度

### 4. 删除痕迹分

依据标题或简介中可能表明删稿/补档行为的文本信号：

- `补档`
- `重传`
- `删稿`
- `应要求删除`
- `限时`

建议：

- 每类 `+10`
- 累计上限 `20`

### 5. 体量修正分

优先中小号，适度抑制过大账号：

- 中小号：`+10`
- 大号：`0`
- 过大号：`-5`

原因：

- 过大号不一定更适合本项目。
- 大号抓取与审核的风控成本通常更高。

### 6. 人工反馈分

人工操作结果必须反映到候选排序：

- 曾被忽略一次：`-15`
- 多次忽略：可累计惩罚
- 被拉黑：不再进入候选池

### 7. 分数分段

前端建议按以下等级展示：

- `80+`：高优先级，优先审核
- `60-79`：值得看
- `40-59`：可观察
- `<40`：默认折叠或不展示

### 8. 评分版本化

评分器必须带 `score_version`，用于：

- 后续调参时区分不同版本结果。
- 支持批量重算候选分数。
- 便于分析“规则调整前后”的命中质量变化。

## API 设计

建议新增以下接口。

### 候选列表

`GET /candidate-creators`

支持参数：

- `status`
- `min_score`
- `keyword`
- `page`
- `page_size`

返回内容：

- 候选基础信息
- 分数
- 关键来源摘要
- 命中原因摘要

### 候选详情

`GET /candidate-creators/:id`

返回内容：

- 候选基础信息
- 来源列表
- 评分明细
- 最近公开视频摘要
- 当前审核状态

### 手动触发发现

`POST /candidate-creators/discover`

行为：

- 立即创建一条 `discover` 任务。
- 返回任务已入队结果。

### 审核动作

- `POST /candidate-creators/:id/approve`
- `POST /candidate-creators/:id/ignore`
- `POST /candidate-creators/:id/block`
- `POST /candidate-creators/:id/review`

行为约束：

- `approve`：写入正式 `creators`，候选改为 `approved`。
- `ignore`：候选改为 `ignored`，记录反馈惩罚。
- `block`：候选改为 `blocked`，后续发现直接过滤。
- `review`：把 `ignored` 候选重新放回 `reviewing`。

### 幂等与事务要求

- `approve` 必须幂等：重复批准同一个候选时，不能重复创建 `creators`，应直接返回已存在结果。
- `approve` 建议在单事务内完成：
  - 校验候选状态
  - Upsert 正式 `creators`
  - 更新候选状态为 `approved`
  - 按配置决定是否创建一次 `fetch` 任务
- `ignore / block / review` 也应做状态校验，避免非法流转。

## 前端交互设计

建议在现有 Web 中新增一个 `候选池` 页面。

### 页面结构

#### 顶部统计

- 新候选数
- 高优先级候选数
- 今日发现数
- 已忽略数

#### 筛选区

- 状态筛选
- 最低分筛选
- 来源类型筛选
- 关键词搜索

#### 列表区

每行展示：

- 博主名称
- UID
- 分数
- 粉丝量
- 最近发现时间
- 命中原因摘要
- 操作按钮：`加入追踪 / 忽略 / 拉黑 / 查看详情`

#### 详情抽屉

展示：

- 来源列表
- 评分明细
- 最近公开视频摘要
- 推荐原因解释

### 审核行为

- 批准成功后，列表即时更新状态。
- 如果配置允许，可在批准后自动触发一次 `fetch`。
- 忽略和拉黑必须给出显式反馈，避免误操作。

## 调度与任务设计

### 新任务类型

在现有任务体系中新增：

- `discover`

这意味着现有 `jobs.type` 枚举需要同步扩展为包含 `discover`。

职责边界：

- 只负责发现候选与打分。
- 不负责下载、不负责检查下架、不直接修改视频状态。
- 失败不影响已有任务类型。

### 默认调度策略

建议默认值：

- 自动发现周期：`24h`
- 支持手动发现按钮即时触发
- 每轮最大关键词数：可配置
- 每个关键词最大搜索页数：可配置
- 每轮最大落库候选数：可配置

### 资源控制

发现链路必须独立限速：

- 独立 QPS 配置
- 独立并发配置
- 每轮总候选上限
- 每个来源博主的扩散上限

## 配置设计

建议新增配置项：

```yaml
discovery:
  enabled: true
  interval: 24h
  max_keywords_per_run: 20
  max_pages_per_keyword: 2
  max_candidates_per_run: 100
  max_related_per_creator: 10
  auto_enqueue_fetch_on_approve: true
  score_version: v1
  keywords:
    - 影视剪辑
    - 高能片段
    - 补档
    - 重传
    - 直播切片
    - 演唱会
    - MV
  score_weights:
    keyword_risk:
      max: 40
    activity_30d:
      low: 5
      medium: 10
      high: 15
    similarity:
      weak: 5
      medium: 10
      strong: 20
    deletion_trace:
      single: 10
      max: 20
    account_size:
      small_bonus: 10
      oversize_penalty: -5
    feedback:
      ignore_penalty: -15
```

要求：

- 发现链路的所有阈值都应可配置。
- 评分配置变更后支持批量重算。
- 配置保存后沿用现有“保存并重启生效”机制。

## 风控边界

候选发现天然比正式追踪更容易触发平台搜索和关系查询，因此必须明确边界：

- 发现任务默认低频执行，不允许高频轮询搜索页。
- 单轮任务要有限制，禁止无限翻页。
- 对异常返回、风控返回、登录异常采用失败快返和退避。
- 发现链路只落库候选，不触发下载，不扩大额外流量。
- 发现失败只影响本轮任务，不能阻塞主任务队列。

## 测试策略

建议分 4 层验证。

### 1. 评分单测

覆盖：

- 各评分因子是否按预期加减分
- 上限和下限是否生效
- 忽略惩罚是否累计
- `score_version` 是否正确写入

### 2. 发现器单测

覆盖：

- 关键词发现去重
- 多来源聚合同一候选
- 已 `approved` 候选再次被发现时不重复建档
- 已 `blocked` 候选再次被发现时被过滤

### 3. 仓储层测试

覆盖：

- 候选表、来源表、评分明细表的增删改查
- 状态流转约束
- 分页、筛选、排序
- 批准后与 `creators` 的去重写入

### 4. API 与前端联调测试

覆盖：

- 候选池列表
- 候选详情
- 批准、忽略、拉黑
- 手动触发发现
- 页面筛选与状态更新

## 实施顺序建议

推荐按以下顺序实现：

1. 数据表与 repo
2. 评分器
3. 关键词发现
4. 候选池 API
5. 前端候选池页面
6. 批准后写入正式 `creators`
7. 关系扩散发现
8. 配置化与调参
9. 观察一段时间后再决定是否做反馈学习

## 第一阶段范围

第一阶段只做以下内容：

- 新增 `candidate_creators`、`candidate_creator_sources`、`candidate_creator_score_details`
- 新增 `discover` 任务类型
- 支持关键词发现
- 支持评分与原因解释
- 支持候选池列表、详情、批准、忽略、拉黑
- 支持批准后写入正式追踪
- 支持基础配置与限速

第一阶段明确不做：

- 多平台发现
- 自动批准
- 机器学习/反馈学习
- 复杂作者图谱
- 大规模实时发现流

## 成功标准

当满足以下条件时，可以认为第一阶段完成：

- 系统每天能稳定产出一批候选博主。
- 每个候选都能解释“为什么会被推荐”。
- 用户可以在前端一键批准并转入正式追踪。
- 发现链路不影响现有抓取、下载、检查、清理链路。
- 所有阈值、频率、关键词和权重均可配置。

## 开放问题

以下问题故意留到实现计划阶段再细化：

- B 站搜索和关系扩散分别复用哪些现有客户端能力，哪些需要新增接口。
- 关系扩散的数据源是否优先使用搜索共现，还是优先使用公开视频列表相似度。
- 批准后是否默认立即入队 `fetch`，还是只在前端提供二次确认。
- 候选池页面是否需要支持批量审核。

## 最终建议

第一阶段把重点放在“闭环稳定性”而不是“发现智能度”上：

- 先保证能发现。
- 再保证能解释。
- 再保证能审核和转正。
- 最后再考虑更复杂的发现优化。

这样更符合当前项目阶段，也更容易在真实运行数据中逐步调参。
