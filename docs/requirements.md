# 需求说明（当前版本）

## 1. 产品目标

本项目当前目标是围绕 B 站博主视频维护，提供一套可长期运行的单机系统，用
于：

- 维护正式追踪博主列表。
- 周期性发现新视频并下载到本地。
- 持续检查已收录视频的可访问状态，沉淀“绝版视频库”。
- 在存储受限时按策略清理普通视频，尽量优先保留绝版内容。
- 通过前端控制台与 HTTP API 提供运维、审核、状态查看和配置修改入口。

## 2. 当前范围

### 2.1 当前已交付范围

- 平台范围：当前只支持 B 站。
- 部署形态：单后端服务 + 独立前端控制台，支持本地运行与 Docker Compose
  部署。
- 数据存储：使用 MySQL 保存业务数据，使用本地文件系统保存视频文件与浏览
  投影。
- 管理入口：
  - 博主文件动态同步（YAML / JSON）
  - HTTP API
  - 前端控制台
- 发现链路：已支持候选池自动发现，包含“关键词发现 + 一跳关系扩散 + 人工
  审核转正”。
- 运维能力：已支持任务查看、单视频查询、单视频重下、单视频检查、手动
  cleanup、系统状态查看、配置在线编辑与保存后重启。

### 2.2 当前非目标

- 多平台采集（抖音、快手、小红书等）。
- 多用户、权限、租户隔离。
- 移动端或桌面端原生 App。
- 云端对象存储、分布式下载集群或多节点调度。
- 候选池自动转正；当前必须人工审核。
- CSV 导入导出流程；当前正式支持的是 YAML / JSON 博主文件和 HTTP 管理。

## 3. 术语

- 博主（Creator）：B 站用户，可通过 UID 管理，也可由名称解析 UID 后建档。
- 候选（Candidate）：由自动发现链路补充、等待人工审核的潜在博主。
- 视频（Video）：博主发布的单个视频。
- 拉取（Fetch）：读取博主最新视频列表并更新视频元数据。
- 下载（Download）：把视频文件保存到本地 `store/`。
- 检查（Check）：检测视频当前是否仍可访问。
- 绝版（Out-of-print）：视频当前不可访问，状态为 `OUT_OF_PRINT`。
- 稳定（Stable）：视频在稳定阈值内多次检查仍可访问，状态为 `STABLE`。
- 浏览投影（Library Projection）：从真实存储派生出来的人工浏览目录
  `library/`，包含符号链接和 `_meta/*.json`。

## 4. 核心需求

### 4.1 博主管理

- 支持通过 `POST /creators` 新增或更新博主。
- 支持查询当前启用中的博主列表。
- 支持通过 `PATCH /creators/{id}` 修改博主名称和状态。
- 支持通过 `DELETE /creators/{id}` 手工移除博主；移除后状态为 `removed`，
  不会被文件同步自动恢复。
- 博主状态至少包含：
  - `active`
  - `paused`
  - `removed`
- 支持通过博主文件（YAML / JSON）动态同步：
  - 新增或更新文件中的博主。
  - 从文件中移除的博主自动停用为 `paused`。
  - 已被手工移除的博主不会因文件再次出现而自动恢复。
- 支持只提供名称添加博主；系统需尝试解析 UID 并缓存解析结果。

### 4.2 候选池自动发现与人工审核

- 系统支持周期性或手工触发 discover 任务。
- 当前自动发现只包含两条入口：
  - 关键词发现。
  - 从已追踪博主出发的一跳关系扩散。
- 候选池需支持查询列表、查看详情、查看来源和评分拆解。
- 候选状态至少包含：`reviewing`、`ignored`、`blocked`、`approved`。
- 已被人工标记为 `ignored` 的候选再次被自动发现时，需保持 `ignored`，只更
  新最近发现时间和相关评分信息，不得自动回退为 `reviewing`。
- 只有人工 `approve` 后，候选才会转为正式追踪博主。
- 若开启 `discovery.auto_enqueue_fetch_on_approve`，批准候选后可自动触发一
  次该博主的定向 fetch；不得触发全量 fetch。

### 4.3 新视频发现、下载与任务系统

- 系统需周期性拉取已启用博主的最新视频列表。
- 新发现且未入库的视频需入库，并创建下载任务。
- 任务系统当前至少支持：
  - `fetch`
  - `download`
  - `check`
  - `cleanup`
  - `discover`
- 相同活动任务需要去重，避免重复入队或重复下载。
- 任务状态至少包含：`queued`、`running`、`success`、`failed`。
- 需要提供最近任务列表、状态与失败原因查询能力。

### 4.4 视频生命周期与状态维护

- 当前视频生命周期以批量调度 + worker 执行为主，不要求为每个视频建立独
  立定时器。
- 当前视频状态至少包含：
  - `NEW`
  - `DOWNLOADING`
  - `DOWNLOADED`
  - `OUT_OF_PRINT`
  - `STABLE`
  - `DELETED`
- 上述 6 个值仍是当前公开状态基线；但下载永久失败时，运行时
  `video.changed` 事件仍可能出现 `FAILED`，前端应把它视为当前 SSE 例外，
  不要反推为公开状态集合已扩展。
- 新视频默认进入 `NEW`，下载成功后进入 `DOWNLOADED`。
- 视频不可访问时标记为 `OUT_OF_PRINT`，并记录时间。
- 视频在稳定阈值内持续可访问时可标记为 `STABLE`。
- 被清理删除后标记为 `DELETED`。
- 启动恢复时，需要修复以下残留状态：
  - `running` 任务重新入队。
  - 无活动下载支撑的 `DOWNLOADING` 视频修复为 `NEW` 或 `DOWNLOADED`。
  - 缺失真实文件的 `DOWNLOADED` 视频回退为 `NEW`。

### 4.5 下架检查

- 系统需按调度周期批量检查视频可访问性，默认目标频率为每天 1 次。
- 已下载和稳定视频都应纳入检查范围。
- 若检查命中不可访问，标记为 `OUT_OF_PRINT`。
- 若超过稳定阈值仍可访问，标记为 `STABLE`。
- 检查链路需支持单视频手工触发。

### 4.6 存储、清理与浏览投影

- 视频真实文件统一保存在 `storage.root_dir/store/`。
- 系统需在 `storage.root_dir/library/` 下维护人工浏览投影目录。
- 投影目录按博主组织，并输出 `_meta/*.json` 供人工浏览。
- 存储超过阈值时，系统需触发 cleanup，把空间释放到安全阈值以下。
- cleanup 当前至少应考虑以下因素：
  - 是否绝版
  - 博主粉丝量
  - 播放量
  - 收藏量
  - 文件大小（可作为同优先级候选的并列打散因素）
  - 保留期
- 绝版内容默认优先保留。
- 应支持手工触发 cleanup。

### 4.7 前端控制台与实时状态

- 系统当前包含独立前端控制台，不再是“仅后端服务”。
- 前端控制台需要支持：
  - 查看博主、任务、视频、系统状态、存储状态。
  - 管理候选池审核流程。
  - 触发 fetch / check / cleanup / discover。
  - 在线读取和编辑配置文件。
- 页面先加载快照，再通过 `/events/stream` 接收 SSE 增量事件。
- 当前事件流至少覆盖：
  - `job.changed`
  - `video.changed`
  - `creator.changed`
  - `storage.changed`
  - `system.changed`
- 前端在断线或后端重启后需要能自动重连，并补一次快照恢复一致性。

### 4.8 配置管理

- 当前运行配置来源于配置文件，而不是数据库配置中心。
- 系统需支持通过 `GET /system/config` 读取当前配置文件内容与路径。
- 系统需支持通过 `PUT /system/config` 写回整份配置文件内容。
- 保存前必须完成 YAML 与业务配置校验。
- 内容未变化时不得触发重启。
- 内容有变化且保存成功后，需要安排后端重启重新加载配置。
- 重启窗口内接口短暂失败属于当前承认的正常现象。

## 5. 非功能性要求

- 单机可运行，支持随时重启、恢复和继续处理任务。
- 关键流程具备幂等性，避免重复入队、重复下载、重复恢复。
- 通过日志、系统状态接口、任务列表和 SSE 事件提供可观测性。
- 调度周期、稳定阈值、存储容量、候选发现和 Cookie 等关键行为可配置。
- 前后端契约应稳定，状态枚举、时间格式和事件类型保持一致。
- 当前实现以实际可维护性优先，不追求多节点或分布式复杂度。

## 6. 约束与假设

- 只处理合法合规可访问的公开内容。
- 默认单租户、单实例部署。
- 当前数据库为 MySQL，不再以 SQLite 作为当前版本前提。
- 浏览投影 `library/`、元数据文件和前端缓存都属于派生产物，不是业务真
  源。
- 当前风控处理依赖 Cookie / SESSDATA 与退避策略，但不保证规避所有平台限
  制。

## 7. 当前验收基线

- 新增或恢复一个正式博主后，系统能在调度周期内发现新视频并创建下载任务。
- 已下载视频在检查周期内能被正确识别为可访问、稳定或绝版。
- 存储超过阈值时，cleanup 能把空间释放到安全阈值以下。
- 候选发现链路能写入候选，人工审核后能转正为正式博主。
- 前端控制台能加载快照、接收 SSE 增量，并在后端重启后恢复一致状态。
- 配置在线保存时，校验失败不会落盘；校验通过且内容变化时会安排重启。
